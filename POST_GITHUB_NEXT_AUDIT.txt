POST-GITHUB - PROCHAIN AUDIT NEURAL STOCK EXCHANGE
==================================================

Statut
------
Le developpement intra-git est considere termine pour l'instant.
Le projet passe en phase post-GitHub.

Cette note n'est pas une liste d'actions a executer maintenant.
Elle sert a preparer le prochain audit complet du depot.


Objectif du prochain audit
--------------------------
Faire une revue complete de Neural Stock Exchange apres publication GitHub afin de definir
ce qui permettra de faire passer le projet a un niveau nettement superieur.

L'audit devra couvrir :

- lisibilite globale du repo pour un lecteur externe ;
- robustesse du pipeline Rev4 ;
- pertinence scientifique de l'evaluation ;
- qualite des rapports et graphiques ;
- coherence des modeles legacy ;
- dette technique restante ;
- risques de surpromesse financiere ;
- possibilites de visualisation interactive.


Axes de progression a etudier
-----------------------------
1. Evaluation Rev4 avancee

- Ajouter une validation walk-forward simple.
- Evaluer separement les periodes de stress.
- Comparer d'autres baselines naives ou statistiques.
- Analyser plus proprement les erreurs par regime.
- Mesurer la valeur du signal directionnel sans le survendre.

2. Signal de stress

- Clarifier le role de `Panic_Mode`.
- Tester plusieurs seuils de volatilite.
- Produire une lecture precision / recall sur les periodes de stress.
- Distinguer prediction de prix et detection de regime.

3. Flux multi-marches

- Etudier Dow Jones + NASDAQ.
- Comparer Dow seul, NASDAQ seul, Dow + NASDAQ.
- Verifier si le multi-marche apporte un signal ou seulement du bruit.

4. Visualisation locale interactive

Comparer les options :

- Streamlit : probablement le meilleur choix pour une interface locale data science rapide.
- Flask : utile seulement si une API ou un backend devient necessaire, pas prioritaire.
- React : interessant pour une experience visuelle plus forte, mais plus couteux a maintenir.

Point a trancher plus tard :
faut-il faire un dashboard local Streamlit centre sur l'analyse Rev4, ou une interface React plus ambitieuse ?

5. Dashboard statique GitHub Pages

Objectif potentiel :
creer un deuxieme dashboard statique publiable sur GitHub Pages.

Pistes :

- dashboard sans backend ;
- lecture de fichiers JSON exportes depuis `reports/` ;
- affichage des metriques, graphiques, verdict critique et contexte Rev4 ;
- positionnement portfolio public ;
- aucune inference en ligne ;
- aucune promesse financiere.

Option probablement solide :
generer un export JSON public depuis le pipeline Rev4, puis construire une app statique React/Vite
deployable sur GitHub Pages.


Questions a traiter au prochain passage
---------------------------------------
- Le dashboard local doit-il etre Streamlit, React ou les deux ?
- Le dashboard GitHub Pages doit-il etre purement statique ou seulement une vitrine visuelle ?
- Quels graphiques doivent etre interactifs ?
- Faut-il garder les modeles `.pt` versionnes ou les documenter comme artefacts historiques regenerables ?
- Quel niveau de documentation faut-il ajouter avant de presenter le repo comme projet portfolio majeur ?
- Est-ce qu'un focus detection de stress est plus fort qu'un focus forecasting de prix ?


Regle de positionnement
-----------------------
Neural Stock Exchange doit rester un projet experimental serieux.

Il peut devenir un laboratoire de forecasting et de detection de regimes financiers,
mais il ne doit pas etre presente comme :

- un bot de trading ;
- un outil de conseil financier ;
- un systeme de prediction fiable des crises ;
- un produit SaaS commercialisable a court terme.

La valeur du projet doit rester :

- experimentation ;
- engineering data ;
- LSTM PyTorch ;
- evaluation critique ;
- reporting reproductible ;
- transparence sur les limites.
